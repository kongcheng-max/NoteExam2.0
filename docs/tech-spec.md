## 1. 系统架构

NoteExam 采用前后端分离的三层架构，核心 AI 能力由 DeepSeek 大模型提供。

```
┌──────────────────────────────────────────────────────────┐
│                        用户端                             │
│  ┌──────────────────────────────────────────────────┐    │
│  │            Web 前端 (React / Vue)             │    │
│  │    笔记上传 · 知识点展示 · 试卷预览 · 在线答题      │    │
│  └───────────────────┬──────────────────────────────┘    │
└──────────────────────┼───────────────────────────────────┘
                       │ HTTPS
┌──────────────────────┼───────────────────────────────────┐
│                后端服务层                                  │
│  ┌───────────────────┴──────────────────────────────┐    │
│  │            API 网关 (Nginx / 云网关)              │    │
│  └───────┬──────────────────────────────┬───────────┘    │
│          │                              │                │
│  ┌───────┴───────────┐    ┌────────────┴────────────┐   │
│  │   核心业务服务      │    │   AI 调度服务            │   │
│  │  (Python FastAPI)  │    │  (Python FastAPI)       │   │
│  │                   │    │                         │   │
│  │  · 用户管理        │    │  · 知识提取 Prompt 编排  │   │
│  │  · 笔记处理        │    │  · 试题生成 Prompt 编排  │   │
│  │  · 试卷管理        │    │  · 答案生成 Prompt 编排  │   │
│  └───────┬───────────┘    └────────────┬────────────┘   │
└──────────┼─────────────────────────────┼────────────────┘
           │                             │
    ┌──────┴──────┐              ┌───────┴────────┐
    │  PostgreSQL │              │  DeepSeek API  │
    │   (数据库)   │              │  (大模型服务)   │
    └─────────────┘              └────────────────┘
```

### 架构说明

- **前端**：负责用户交互，所有 AI 调用通过后端中转，前端不直接调用 DeepSeek API；框架可选 React 或 Vue
- **核心业务服务**：处理笔记存储、试卷 CRUD、用户管理等业务逻辑
- **AI 调度服务**：封装与 DeepSeek 的交互，管理 Prompt 模板、Token 控制、错误重试
- **数据库**：存储用户数据、笔记内容、知识点、试卷、答案

## 2. 技术选型

### 选型原则

1. **API 优先使用国内厂商**，降低网络延迟和政策风险
2. **同一厂商优先**，减少供应商管理成本
3. **开源优先**，降低许可证成本

### 前端

| 选型 | 方案 | 理由 |
|------|------|------|
| 框架 | **React (Next.js) / Vue (Nuxt)** | React 生态成熟，Vue 上手更友好；均支持 SSR |
| UI 组件库 | **shadcn/ui / Ant Design Vue** | 分别对应 React 和 Vue 生态的主流组件库 |
| 状态管理 | **Context+SWR / Pinia** | MVP 阶段轻量方案即可 |
| 部署 | **阿里云 ECS / 腾讯云轻量服务器** | 国内云服务商，备案便捷 |
| PWA | **Workbox + manifest.json** | 渐进式 Web 应用，零安装跨平台，离线可用 |

### 后端

| 选型 | 方案 | 理由 |
|------|------|------|
| 语言 & 框架 | **Python + FastAPI** | AI 生态最好，DeepSeek 官方 SDK 支持；高性能异步框架 |
| API 文档 | **自动生成 OpenAPI (Swagger)** | FastAPI 内置，零额外成本 |
| 数据库 | **PostgreSQL** | 成熟的关系型数据库，支持 JSON 字段适合存储试卷结构 |
| 数据库托管 | **阿里云 RDS / 腾讯云 PostgreSQL** | 国内云厂商，运维托管 |
| 缓存 | **Redis**（云服务商托管） | 用于 Prompt 缓存、会话管理 |

### AI 服务

| 选型 | 方案 | 理由 |
|------|------|------|
| 大语言模型 | **DeepSeek API**（深度求索） | 国内头部大模型，中文理解能力强，性价比高 |
| OCR 识别（V1.1） | **百度智能云 OCR** | DeepSeek 暂无 OCR 产品；百度 OCR 是国内最成熟的文字识别服务，识别准确率高 |

> **关于"同一厂商优先"原则**：核心模型 DeepSeek 来自深度求索，但其产品线目前不含 OCR 和云基础设施服务。因此 OCR 推荐百度智能云（国内 OCR 市场份额第一），数据库与部署推荐阿里云或腾讯云，均为国内头部服务商。

### 存储

| 选型 | 方案 | 理由 |
|------|------|------|
| 文件存储 | **阿里云 OSS / 腾讯云 COS** | 用于存储用户上传的图片、PDF |
| 文本存储 | **PostgreSQL** | 笔记文本、试卷内容直接存数据库 |

## 3. DeepSeek 集成

### 3.1 模型选择

| 场景 | 推荐模型 | 理由 |
|------|----------|------|
| 知识点提取 | DeepSeek-V3 | 长文本理解能力强，适合分析整篇笔记 |
| 试题生成 | DeepSeek-V3 | 指令遵循能力强，出题格式稳定 |
| 答案生成 | DeepSeek-V3 | 与出题使用同一模型，保持风格一致 |
| 轻量任务（如格式校验） | DeepSeek-V2.5 或轻量模型 | 降低延迟和成本 |

### 3.2 调用流程

```
用户上传笔记
     │
     ▼
┌─────────────┐
│ 文本预处理    │  ← 去除噪声、分段、Token 估算
└──────┬──────┘
       │
       ▼
┌─────────────────┐
│ Prompt 1: 知识提取 │  ← 输入：笔记全文 + 知识提取指令
│ → DeepSeek API    │  ← 输出：结构化知识点列表 (JSON)
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│ Prompt 2: 试题生成 │  ← 输入：知识点列表 + 题型 + 难度 + 数量
│ → DeepSeek API    │  ← 输出：试卷 (JSON，含题干/选项/答案/解析)
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│ 格式校验 & 组装   │  ← 校验 JSON 结构、拼装最终试卷
└─────────────────┘
```

### 3.3 Prompt 策略

- **知识提取 Prompt**：要求模型以 JSON 格式输出，包含知识点名称、类别、简要描述
- **试题生成 Prompt**：明确指定题型、难度、数量、输出格式；要求题干清晰、选项互斥、答案准确
- **解析生成 Prompt**：要求 1–3 句话，点明核心原理或常见误区

### 3.4 Token 估算

| 环节 | 输入 Token | 输出 Token | 备注 |
|------|-----------|------------|------|
| 知识提取 | ~2,000–8,000 | ~500–2,000 | 取决于笔记长度 |
| 试题生成 | ~1,000–3,000 | ~2,000–8,000 | 取决于题目数量和题型 |
| 合计 | ~3,000–11,000 | ~2,500–10,000 | 单次生成约消耗 10K–20K Token |

## 4. 数据流

### 4.1 核心数据流

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  笔记文本  │───▶│ 知识提取   │───▶│ 试题生成   │───▶│ 格式组装   │───▶│ 试卷展示   │
│  (上传)    │    │ (AI)      │    │ (AI)      │    │ (后端)    │    │ (前端)    │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
      │               │               │               │               │
      ▼               ▼               ▼               ▼               ▼
  笔记存储到       知识点存储到     试题存储到      试卷存储到      用户浏览/
  数据库           数据库          数据库          数据库          答题
```

### 4.2 核心数据表结构（概要）

```
notes (笔记表)
  ├── id, content, type(text/image/pdf), user_id, created_at

knowledge_points (知识点表)
  ├── id, note_id, name, category, importance, created_at

exams (试卷表)
  ├── id, note_id, title, config(json), created_at

questions (试题表)
  ├── id, exam_id, type, difficulty, content(json), answer, explanation, order
```

## 6. 安全与隐私

### 数据传输

- 全链路 HTTPS 加密
- API 密钥（DeepSeek API Key）仅存储在服务端，前端不可见
- 数据库连接使用 SSL

### 数据存储

- 用户笔记内容加密存储
- 数据库访问权限最小化，按服务划分账号
- 定期自动备份数据库

### 隐私保护

- 用户上传的笔记内容**不用于**模型训练（在 DeepSeek API 调用中设置 `do_not_train` 参数）
- 用户可随时删除自己的笔记和生成的试卷
- 不收集不必要的用户个人信息

### 内容安全

- 输入侧：对用户上传的文本内容进行合规审核（接入国内内容安全 API）
- 输出侧：对 AI 生成的试卷内容进行敏感词过滤

## 7. 容错与监控

### 容错策略

| 场景 | 策略 |
|------|------|
| DeepSeek API 超时 | 自动重试 2 次，超出后返回友好提示 |
| DeepSeek API 限流 | 排队机制，提示用户预估等待时间 |
| 生成结果格式异常 | JSON 校验 + 回退重试，失败后人工兜底提示 |
| 数据库异常 | 主备切换（云服务商托管自动处理） |

### 监控指标

| 指标 | 说明 |
|------|------|
| API 调用成功率 | DeepSeek API 调用成功 / 总调用 |
| 平均生成时间 | 从提交到试卷展示的端到端耗时 |
| 格式校验通过率 | 生成结果一次通过 JSON 校验的比例 |
| 用户满意度 | 答题后评分 |

## 8. V1.1 新增能力

### 8.1 OCR 集成（图片上传）

#### 服务选型

选用 **百度智能云 OCR**，理由：
- 国内 OCR 市场份额第一，准确率最高
- 支持通用文字识别、手写体识别、表格识别
- 提供 Python SDK，与后端技术栈一致
- DeepSeek 暂无 OCR 产品，百度为最优国内替代

#### 调用流程

```
用户上传图片
     │
     ▼
┌─────────────┐
│ 图片预处理    │  ← 压缩（≤ 4MB）、旋转矫正
└──────┬──────┘
       │
       ▼
┌─────────────────┐
│ 百度 OCR API     │  ← 通用文字识别 / 手写体识别
│ → 返回识别文本    │
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│ 文本后处理       │  ← 去噪、分段、合并多图
└──────┬──────────┘
       │
       ▼
  进入 DeepSeek 知识提取流程
```

#### 技术要点

| 要点 | 说明 |
|------|------|
| API | 百度智能云「通用文字识别」或「高精度版」；手写笔记用「手写文字识别」 |
| 并发 | 多图并发上传，单次最多 10 张 |
| 预处理 | 前端压缩图片至 ≤ 4MB，后端做旋转/亮度矫正 |
| 结果合并 | 多图识别结果按上传顺序拼接 |
| 成本 | 百度 OCR 每月免费额度 1,000 次，超出后按量计费 |

### 8.2 PDF 解析

#### 双路径处理

```
用户上传 PDF
     │
     ├── 文字型 PDF ──▶ PyMuPDF 直接提取文本 ──▶ 进入知识提取
     │
     └── 扫描型 PDF ──▶ 逐页转图片 ──▶ 百度 OCR ──▶ 合并文本 ──▶ 进入知识提取
```

#### 技术要点

| 要点 | 说明 |
|------|------|
| 文字型 | 使用 PyMuPDF (fitz) 提取文本，保留段落结构 |
| 扫描型 | 使用 pdf2image 逐页转 PNG，再调用百度 OCR |
| 自动判断 | 优先尝试文本提取；若提取结果为空或字符过少，自动切换 OCR |
| 页数限制 | 建议 ≤ 50 页；超限时提示用户分次上传 |
| 异步处理 | PDF 解析为异步任务，前端轮询进度 |

### 8.3 试卷导出

#### 方案

后端将试卷 JSON 渲染为 HTML 模版 → 生成可打印文件。

| 要点 | 说明 |
|------|------|
| 导出格式 | 优先支持 PDF（通用）、后续可扩展 Word |
| 生成方式 | 后端 Python 使用 WeasyPrint / Playwright 将 HTML 转为 PDF |
| 排版 | 保留题型标签、难度标签、题目序号；留答题空白 |
| 含答案版 | 导出时可选「含答案与解析」或「仅试题」 |
| 性能 | 20 题以内的试卷导出时间 ≤ 5 秒 |

### 8.4 新题型 Prompt 策略

**多选题**：除正确选项外，要求生成 2–3 个合理干扰项（基于常见误区）。

### 8.5 V1.1 架构更新

V1.1 在 MVP 架构基础上新增 OCR 服务和文件存储：

## 9. V1.2 新增：PWA 落地

### 9.1 目标

将现有 Web 前端封装为 PWA（Progressive Web App），用户无需应用商店即可"安装"到桌面和移动主屏幕，获得类原生 App 体验。一套代码同时服务 Web 和 App 场景。

### 9.2 技术方案

```
┌─────────────────────────────────────────────────┐
│                    PWA 层                         │
│                                                  │
│  ┌──────────────┐  ┌──────────────┐              │
│  │ manifest.json│  │Service Worker│              │
│  │ (App 元数据)  │  │ (离线 + 缓存) │              │
│  └──────────────┘  └──────┬───────┘              │
│                           │                      │
│                   ┌───────┴────────┐             │
│                   │  Workbox       │             │
│                   │ (SW 工具链)     │             │
│                   └───────┬────────┘             │
└───────────────────────────┼──────────────────────┘
                            │
         ┌──────────────────┴──────────────────┐
         │           现有 Web 前端               │
         │     React (Next.js) / Vue (Nuxt)     │
         └──────────────────────────────────────┘
```

| 组件 | 选型 | 说明 |
|------|------|------|
| App 清单 | `manifest.json` | 定义 App 名称、图标、启动 URL、主题色、显示模式 |
| Service Worker | **Workbox** (Webpack/Vite 插件) | Google 维护的 SW 工具库，自动预缓存和运行时缓存 |
| 离线策略 | Cache First (静态) + Network First (API) | 静态资源优先缓存，API 请求网络优先回退缓存 |
| iOS 适配 | `apple-touch-icon` + `<meta>` 标签 | iOS Safari 对 PWA 支持有限，需额外处理 |

### 9.3 核心配置

#### manifest.json

```json
{
  "name": "NoteExam - AI 智能出题",
  "short_name": "NoteExam",
  "description": "将学习笔记自动转化为试卷",
  "start_url": "/",
  "display": "standalone",
  "background_color": "#ffffff",
  "theme_color": "#4F46E5",
  "icons": [
    { "src": "/icons/icon-192.png", "sizes": "192x192", "type": "image/png" },
    { "src": "/icons/icon-512.png", "sizes": "512x512", "type": "image/png" },
    { "src": "/icons/icon-512-maskable.png", "sizes": "512x512", "type": "image/png", "purpose": "maskable" }
  ]
}
```

#### Service Worker 缓存策略

| 资源类型 | 策略 | 说明 |
|----------|------|------|
| HTML / CSS / JS / 字体 | **Cache First** | 构建产物带 hash，永久缓存，首次访问后秒开 |
| 静态图片 / 图标 | **Cache First** | 预缓存，离线也能正常显示 |
| API 请求 | **Network First** | 在线时实时获取，超时 5 秒后回退到缓存数据 |
| 图片/PDF 上传 | **Network Only** | 上传操作不走缓存 |
| 试卷生成请求 | **Network Only** | 必须在线，失败给出明确提示 |

### 9.7 注意事项

- PWA 与现有 Web 前端**零冲突**，不影响现有 Web 用户
- `manifest.json` 和 Service Worker 对未安装用户完全透明
- iOS 推送通知需后续配合原生 App 实现，PWA 阶段不覆盖
- 需服务器支持 HTTPS（已具备）
- Service Worker 仅在 `localhost` 和 HTTPS 下生效，开发环境注意区分
